Pip freshness redesign

Stop refreshing on every open. Sync when there is a reason.

The current system already has Plaid webhooks, sync jobs, stale markers, and same-day Spendable Cash Today math. The plan is to connect those pieces into one app-open decision instead of treating every launch like a refresh event.

Recommended implementation shape

Use existing plaid_webhook_events and pip_sync_jobs as the durable state. App open reads that state, runs sync only for pending bank updates or stale data, and otherwise lets Pip say useful guidance.

pip_sync_jobs

Missing bridge to the app

App-open only sees that a job is pending, not that it came from a bank update.

The state management idea

State source

No new app-wide state table required

The durable state can be derived from existing rows: active pip_sync_jobs, recent plaid_webhook_events, and connected_institutions.last_successful_sync_at.

New concept

Bank update cue

Add a small typed object to the app-open/current-state APIs: none, webhook_pending, syncing, stale, needs_repair, or recent.

User effect

Pip speaks only when true

Pip says bank-update copy only when Plaid actually sent a signal. Otherwise the opening bubble can be normal coaching, same-day spend, tight-day guidance, or a freshness note.

Proposed flow

1. Plaid

Update signal

Plaid sends a webhook when synced transaction data has changed.

2. Backend

Record and queue

Pip records the webhook and creates a plaid_webhook sync job.

3. App open

Read cue

The app-open route checks for active webhook jobs before doing any sync work.

4. Sync

Pull only when warranted

If there is a pending bank update or stale data, run /transactions/sync.

5. UI

Explain the truth

Pip says what happened and shows the refreshed Spendable Cash Today if it changed.

Decision table for app open

Input state Backend action User-facing copy
Active pip_sync_jobs.reason = "plaid_webhook" Run or process that job immediately, then reload Pip Cash. "I found bank updates. Let me pull them in."
Recent sync, no active webhook job Do not call Plaid. Use cached snapshot. Normal guidance, same-day spend, tight-day note, or "last updated..."
Last successful sync older than threshold Run ordinary /transactions/sync as a stale-data check. "Your bank data is getting old, so I am checking it now."
Pending or running sync job Do not start a duplicate job. Poll/reload freshness state. "Your bank update is already syncing."
Repairable Plaid error Do not sync. Route to account repair. "This connection needs a quick reconnect before I can update it."

Implementation map

Backend changes
  • 1 Expand the pending sync job loader so app-open can see job reason, provider, institution, source webhook, and status.
  • 2 Add a small getBankUpdateCueForUser helper that returns the typed cue app-open and current snapshot need.
  • 3 Change getAppOpenSyncDecision from a 60 second guard to webhook-first, stale-second, otherwise skip.
  • 4 Make webhook jobs process promptly: either app-open claims them immediately or the scheduled processor runs pending jobs even when scheduled enqueue is off.
UI changes
  • 1 Replace the current string-only app-open message with a typed status payload.
  • 2 Update the opening bubble planner so refresh copy only wins when the typed cue says refresh work is real.
  • 3 Keep existing same-day spend priority after a sync. That is the moment Pip can say it found spending and took it off today.
  • 4 Show "last updated" as a receipt, not an excuse, whenever no sync is needed.
