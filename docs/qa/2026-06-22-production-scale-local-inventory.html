Run Rules

Boundaries for keeping the inventory production-like without using sensitive or destructive surfaces.

Execution Boundary
  • Local data only: use PIP_SUPABASE_MODE=off, PIP_LOCAL_FAKE_APP_MODE=1, and scenario=production-scale for financial data.
  • Production-like runtime: verify with npm run build and npm run start, not only next dev.
  • Browser evidence: use Codex in-app Browser with backend iab.
  • Safety: no production writes, no real provider tokens, no real user data, no destructive account/data deletion action without explicit approval.
  • Rerun policy: after any fix, rerun the affected focused check, then rerun the complete inventory gates.

Data Fixture Acceptance

The synthetic scenario has to be large enough to expose production-scale behavior while staying sanitized and deterministic.

Synthetic Fixture

Acceptance

  • isFakeDataScenario("production-scale") returns true.
  • /api/pip-cash?scenario=production-scale works when Supabase is disabled.
  • Contains at least 8 accounts across checking, savings, credit card, and loan.
  • Contains at least 500 transactions across 18 months.
  • Includes income, rent, purchases, transfers, card payments, refunds, fees, pending transactions, matched card payments, and an unmatched card-payment warning path.
  • Contains no emails, OAuth/provider names, production project names, full account numbers, or Tyler-specific identifiers.
  • Produces a valid calculatePipCash() result without throwing.

Finite Edge Cases

  • Current-month partial data, same-day posted purchase, and same-day pending duplicate purchase.
  • Pending current-month income and pending travel purchase.
  • Matched and unmatched credit-card settlement rows.
  • Refund and bank-fee months.
  • Multiple institutions and multiple account kinds.
  • Protected savings plus separate reserve savings.
  • Loan account present but excluded from Pip Cash.
  • Active savings goal included in Spendable Cash Today.

Bug Log

Every bug logged during the run, with reproduction evidence, shared-cause diagnosis, fix, and regression hook.

Fixed Findings
DATA-001Fixed

Synthetic local data

Evidence
npx vitest run src/lib/fake-data.test.ts first failed because production-scale was not a recognized scenario.
Root cause
No production-scale local scenario existed.
Fix
Added deterministic production-scale snapshot in src/lib/fake-data.ts.
Regression
src/lib/fake-data.test.ts validates size, account/transaction coverage, sanitization, and calculatePipCash().
DATA-002Fixed

Synthetic local data

Evidence
In-app Browser showed $0 for /app?scenario=production-scale; local API showed v2 state shortfall.
Root cause
The current-month fixture had a zero-amount payroll and a same-day pending travel hold.
Fix
Made current-month payroll nonzero, scaled current-month discretionary rows, and moved the pending travel hold off the as-of date.
Regression
src/lib/fake-data.test.ts requires nonzero transactions, positive v2 Spendable Cash Today, and non-shortfall state.
LOCAL-001Fixed

Production-like local /app

Evidence
In-app Browser hit http://127.0.0.1:3001/app?scenario=production-scale and saw "Pip access is temporarily unavailable" instead of the fake app.
Root cause
NODE_ENV=production disables dev onboarding/demo routes, and Supabase-off was treated only as unavailable.
Fix
Added explicit PIP_LOCAL_FAKE_APP_MODE=1 local app shell and rejected that flag in beta deployment checks.
Regression
src/app/app/page.test.tsx covers the local fake app route; scripts/check-deployment-env.test.ts blocks beta mode with the flag.
LOCAL-002Fixed

Production-like local /api/agent

Evidence
npm run eval:agent:major against the built local server returned HTTP 503 for every case with agent-model-gate-unavailable.
Root cause
Production mode disables the in-memory model gate, and Supabase-off mode has no RPC-backed gate.
Fix
Allowed the in-memory gate only when PIP_LOCAL_FAKE_APP_MODE=1; beta deployment checks reject that flag.
Regression
src/lib/agent/agent-model-gate.test.ts covers production + Supabase-off + local fake mode.
AGENT-001Fixed

Major agent dogfood

Evidence
OpenAI-backed expanded evals exposed response-contract failures: card replies ended with questions, no-card replies promised cards/views, and read-only/card-provider wording caused false positives.
Root cause
Model output was stochastic and the visible-response guard/scorer did not cover several production-language variants.
Fix
Strengthened visible-response-guard.ts, deterministic response-mode normalization, and eval scorer exceptions.
Regression
src/lib/agent/ai-agent.test.ts, visible-response-guard.test.ts, scripts/eval-agent.test.ts; primary, expanded, and multi-turn OpenAI suites passed.
GATE-001Fixed

Pip money companion gate

Evidence
npm run test:pip-money-companion-gate initially blocked on "No default verification is wired for DOGFOOD-001".
Root cause
The default gate adapter had deterministic category commands but no sanitized local DOGFOOD path.
Fix
Added a fail-closed local sanitized DOGFOOD adapter that verifies /app plus /api/pip-cash?scenario=production-scale.
Regression
scripts/pip-money-companion-gate.test.ts; /tmp/pip-money-companion-gate-rerun/manifest.json passed the full gate.
DATA-003Fixed

Production-scale fixture

Evidence
DOGFOOD-002 failed because production-scale had no same-day discretionary spend on the as-of date; DOGFOOD-005 would not cover savings goal impact.
Root cause
Fixture breadth was large but missed two user-facing workflow edges.
Fix
Added same-day posted+pending duplicate purchase and an active savings goal included in Spendable Cash Today.
Regression
src/lib/fake-data.test.ts; full money-companion gate passed.
API-001Fixed

Hydrated /app?scenario=production-scale chat

Evidence
In-app Browser submitted "Can I spend $50?" and the UI showed "I couldn't answer that cleanly."
Root cause
/api/agent request schema omitted production-scale, so the hydrated client sent a scenario that the route rejected.
Fix
Added production-scale to the agent route schema.
Regression
src/app/api/agent/route.test.ts; in-app Browser rerun returned a purchase simulation card.

Final Status

Final state is a clean local pass for sanitized production-scale data under production-like settings. No production writes, real provider data, real user financial data, or destructive actions were used.

The local production-like server used the approved OpenAI key loaded from the canonical local env without printing or copying the key. Real connected-account DOGFOOD remains intentionally outside this sanitized local pass unless explicitly approved at action time.
